# 里程碑 M1：数据源与数据模型管理

> 状态：**方案评审中**（待用户审核通过后拆分子 issue）
> 基线：dev（`core.Report` / 顶层 `dto` 包 / 前端 `report-datasource` 已并入 `report-engine`）

## 1. 背景与目标

当前报表引擎已具备「设计 + 渲染 + 配置持久化」全链路，但**数据从哪来**这一层仍是 example 写死的 CSV（`DataModelSeeder`）。本里程碑补齐**数据接入与建模**的完整管理能力，让用户能在界面上：

1. 配置数据类型容器（驱动 / 存储位置）
2. 配置具体数据源 + 数据集（连库取元数据 / 上传 Excel）
3. 组装数据模型（选数据集 + 建合集 + 配关系）供报表引用

### 跨阶段总原则

- **管理能力全部由 `report-engine` 包提供组件**，`app-pc` 只做路由挂载 + 注入 API，不写业务 UI。
- **界面风格与报表设计器（`ReportEngine`）一致**：顶部操作菜单 + 标题栏，内容区用 antd `Table` 分页呈现，弹窗/抽屉拆独立组件（遵循 CLAUDE.md 前端规范）。
- **只保留 DB / EXCEL / CSV 三种数据源类型**（API/JSON 已在前序重构删除）。CSV 仅作内置/示例，不在用户可创建类型内。
- **数据模型不再有「参数」页签**（报表参数在报表配置阶段管理）。

### 开发优先级（严格按序，前者是后者前置）

```
阶段一 数据类型配置管理  →  阶段二 数据集管理  →  阶段三 数据模型界面
```

## 2. 阶段一：数据类型配置管理

### 2.1 目标

- **DB**：以驱动划分（如 MySQL-8 / PG-15 各一条）——人工上传驱动 jar + 选驱动类，**可建多条**，是「数据类型配置管理」界面的主体。
- **EXCEL / CSV**：**不需人工维护存储位置**，统一走系统参数的默认存储目录（见 2.2）。人工只需添加 DB 驱动。

### 2.2 系统参数 `ReportProperties`（基础设施，先做）

把 starter 现有 `ReportFontProperties`（`@ConfigurationProperties("codingapi.report.fonts")`，单字段 `dir`）**重构为全局 `ReportProperties`**（前缀 `codingapi.report`），下挂四个子配置（各含 `dir` + 默认值），作为 `ReportProperties` 的私有属性：

| 子属性 | 配置键 | 默认值 | 用途 |
|---|---|---|---|
| `font`（`FontProperty`） | `codingapi.report.font.dir` | 仅内置 | 自定义字体目录（原 `fonts.dir` 迁移） |
| `driver`（`DriverProperty`） | `codingapi.report.driver.dir` | `./data/drivers` | 驱动 jar 存储目录 |
| `excel`（`ExcelProperty`） | `codingapi.report.excel.dir` | `./data/excel` | Excel 上传存储目录 |
| `csv`（`CsvProperty`） | `codingapi.report.csv.dir` | `./data/csv` | CSV 上传存储目录 |

- 这些目录最终**传递给 framework 作为默认存储路径**使用（jar 加载、excel/csv 落盘读取）。
- ⚠️ 字体配置键 `codingapi.report.fonts.dir` → `codingapi.report.font.dir`（fonts→font），同步改 `ReportEngineAutoConfiguration` 装配与 README。

### 2.3 后端设计（数据类型配置 = DB 驱动）

| 层 | 内容 |
|---|---|
| 实体 | `DataSourceTypeConfig { id, name, DataSourceType type, createTime, updateTime }`（framework `data.datasource`），复用现有 sealed `DataSourceType`。**人工创建的只有 `DbDataSourceType{jarFile, driverClass}`**；Excel/Csv 类型由 `ReportProperties` 默认目录派生，不落用户记录 |
| DTO | `DataSourceTypeDTO { id, name, kind, jarFile, driverClass, createTime, updateTime }`（扁平，`dto.datasource`） |
| 仓库 | `DataSourceTypeRepository`（framework 接口）+ `InMemoryDataSourceTypeRepository`（example） |
| REST | `DataSourceTypeController` `/api/datasource-types`：分页列表 / 详情 / 保存 / 删除（当前主要承载 DB 驱动） |
| jar 上传 | `POST /api/datasource-types/driver-jar`（multipart）→ 存到 `ReportProperties.driver.dir` → 扫描 jar 内 `java.sql.Driver` 实现（优先读 `META-INF/services/java.sql.Driver`，否则扫类名）→ 返回 `{ jarFile, driverClasses[] }` 供前端选择 |
| 驱动加载 | `DriverLoader`（starter 组件）：启动时（`ApplicationReadyEvent`）用 `URLClassLoader` 加载 `driver.dir` 下全部 jar；按每个 DB 配置的 `driverClass` 实例化并 `DriverManager.registerDriver(new DriverShim(driver))`（`DriverShim` 绕过 DriverManager「只信任系统类加载器」的限制）；新增 jar 时热注册 |

> **设计决策（驱动加载范围）**：本阶段**做完整 classLoader 注册**（对齐「启动时加载外部驱动到 classLoader」）。但「用驱动真正连库取数」在**阶段二**才被 `DbDataExtractor` 调用验证——本阶段只保证「驱动可被加载注册」，连库链路阶段二一起跑通。

### 2.4 前端设计（report-engine）

- 组件 `DataSourceTypeManager`（`components/datasource/`）：顶部操作栏（新建驱动）+ antd `Table` 分页（列：名称 / 类型 / 驱动类 / 操作）+ 新建/编辑 `Modal`：上传 jar（`onUploadDriverJar` 回调返回候选驱动类）→ 选驱动类 → 命名。
- report-api：`listDataSourceTypes(current,pageSize)` / `getDataSourceType(id)` / `saveDataSourceType(dto)` / `deleteDataSourceType(id)` / `uploadDriverJar(file)`。
- app-pc：`/datasource-types` 路由 + 菜单入口，仅挂组件。

## 3. 阶段二：数据集管理

### 3.1 目标

全局管理「数据源连接 + 其下数据集」，供数据模型引用。流程：**先选类型 → 配具体连接 / 上传文件 → 解析元数据结构**。三种类型都要支持：

- **DB**：选依赖的 DB 驱动配置（阶段一）→ 配 url / 账号 / 密码 / schema → JDBC 查表与字段。
- **EXCEL**：上传 Excel 文件（落到 `ReportProperties.excel.dir`）→ **每个 sheet = 一个数据集**（首行列名、其余为数据）。
- **CSV**：上传 CSV 文件（落到 `ReportProperties.csv.dir`）→ **单个数据集**（首行列名、其余为数据）。CSV 格式比 Excel 简单（无多 sheet），但同样要做。

### 3.2 后端设计

| 层 | 内容 |
|---|---|
| 实体 | `DataSource`（已是聚合根，持有 `List<Dataset>`）扩展：`typeConfigId`（DB 引用阶段一驱动配置）+ `config`（DB: url/username/password/schema；EXCEL/CSV: 上传文件引用）。`TableDataset` 已聚合 `DataSource` |
| 仓库 | 新增 `DataSourceRepository`（framework，全局数据源/数据集）+ `InMemory…`（example） |
| REST | `DataSourceController`（已存在，扩展为 CRUD）：保存 / 列表 / 详情 / 删除 / 连接测试（已有）/ **元数据解析** |
| 元数据解析 | `POST /api/datasources/{id}/introspect`：DB → JDBC `DatabaseMetaData`/`ResultSetMetaData` 列出表 + 字段（name/dataType/primaryKey）；EXCEL → 每个 sheet 一个 `TableDataset`；CSV → 单个 `TableDataset`（均首行列名、类型默认 STRING） |
| 文件上传 | `POST /api/datasources/upload`（multipart，区分 EXCEL/CSV）→ 存到 `excel.dir` / `csv.dir` → 解析数据集 |

> DB 取元数据复用阶段一注册的外部驱动（`DriverLoader`）。本阶段把「外部 jar 驱动 → JDBC 连库 → introspect」整条链路跑通；同时落地 EXCEL（多 sheet）与 CSV（单集）的上传 + 解析。

### 3.3 前端设计（report-engine）

- 组件 `DataSourceManager`：列出数据源；新建向导——① 选类型（DB 驱动 / EXCEL / CSV）→ ② 配置（DB: url/账密/schema；EXCEL/CSV: 上传文件）→ ③ 解析数据集（introspect 预览：Excel 多 sheet / CSV 单集）→ ④ 保存。
- report-api：`listDataSources` / `saveDataSource` / `deleteDataSource` / `introspectDatasets` / `uploadDataFile`（excel/csv）/ `testConnection`（部分已有）。
- app-pc：`/datasources` 路由 + 菜单入口。

## 4. 阶段三：数据模型界面

### 4.1 目标与布局

数据模型 = 选数据集 + 建合集 + 配关系，供报表引用。**全部组件在 report-engine，app-pc 仅引用，风格对齐 `ReportEngine`**。

```
┌─────────────────────────────────────────────┐
│ 顶部：操作菜单 + 标题栏（对齐报表配置界面）        │
├─────────────────────────────────────────────┤
│ [数据集] [数据合集] [关系]   ← 三个页签           │
│ 每个页签：antd Table 分页呈现                    │
└─────────────────────────────────────────────┘
```

### 4.2 功能

1. **数据模型列表页（前置）**：列出全部数据模型（已有 `DataModelMgmtController` 分页），支持创建（名称等）→ 进入设计页（`/datamodels/{id}`）。

2. **数据集页签**：添加数据集 = 选**全局已配数据集**。选择为**二级联动**：先选数据源 → 再选数据集；一个数据源下多数据集**可多选**。模型只存 `datasetId` 引用，load 时从 `DataSourceRepository` 解析。

3. **数据合集页签（Union）**：创建合集，界面上下两部分：
   - **上**：合集字段定义（顺序列表）+ 合集名称/别名。
   - **下**：选左右两张表（可切换当前模型已选数据集，展示全部字段），配置字段映射。左右两侧字段**按顺序对应合集字段**（数量 = 合集字段数；类型/属性须与合集一致；字段名以合集为准）。
   - → 后端 `UnionDataset { fields, members[{datasetId, mapping}] }`（两个 member，`mapping` = 合集列名→成员实际字段名）。

4. **关系页签**：跨数据集 JOIN 关系编辑（复用/重构现有 `RelationEditor`），表格化 + 分页。

5. **移除「参数」页签**（报表参数归报表配置）。

### 4.3 后端设计

- `DataModel` 已领域化（`datasets` 引用 + `relationships` + `UnionDataset` 内联）。`DataModelMgmtController` 已有 CRUD。
- 数据集来源 = 全局 `DataSource`/`Dataset`（阶段二）；模型存引用、load 时解析。
- `UnionDataset` 已支持 members + mapping，无需结构改动，补前端编辑器即可。
- 清理可能残留的参数管理端点/字段。

### 4.4 前端设计

- 重构现有 `components/data-model/` 为：列表页 + 设计页（三 tab）。
- 数据集 tab：二级联动选择器（数据源 → 数据集多选）。
- 数据合集 tab：`UnionEditor`（上字段定义 + 下左右表字段映射）。
- 关系 tab：表格化 `RelationEditor`。
- 删除 `param-manager`/`param-modal`（数据模型侧）。

## 5. 验收标准

- [ ] 能在界面创建 DB 类型（上传 MySQL/PG 驱动 jar、选驱动类）与 EXCEL 类型（存储路径）。
- [ ] 能创建 DB 数据源（连真实库取表/字段）与 EXCEL 数据源（上传 → sheet 成数据集）。
- [ ] 能创建数据模型：选数据集、建跨源 Union 合集、配关系；保存后报表可引用并渲染。
- [ ] 全部管理 UI 在 report-engine，app-pc 仅路由挂载；风格与设计器一致。
- [ ] 后端 framework/excel 测试全绿；前端 `pnpm build` + 单测全绿。

## 6. 风险与开放问题

1. **外部 JDBC 驱动加载**（最高风险）：`URLClassLoader` + `DriverShim` + `DriverManager.registerDriver` 是标准解法，但需处理多 jar、驱动重复注册、连接关闭后驱动卸载等；建议先支持 MySQL/PG/H2 验证。
2. **Excel 字段类型推断**：本期 spec 只要求「首行列名、其余数据」，类型默认 STRING；是否需按列采样推断类型留待后续。
3. **数据源凭证安全**：沿用现状（明文存内存 + 出口脱敏，落盘加密交使用方仓库）。
4. **CSV 已纳入用户可创建类型**（阶段二上传解析，单数据集）；example 现有内置 CSV 数据模型（`DataModelSeeder`）保持不变，与用户上传 CSV 并存。
